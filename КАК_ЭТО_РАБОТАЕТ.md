# Как работает мессенджер (по принципам Delta Chat)

## Ключевые улучшения из Delta Chat

### 1. UIDNEXT вместо UNSEEN
**Проблема UNSEEN:**
- Зависит от флага \Seen
- Другой клиент может пометить письмо → оно исчезнет из UNSEEN
- Нужно искать по всему ящику (медленно)

**Решение UIDNEXT (Delta Chat):**
- Гарантированный счётчик новых писем
- Не зависит от флагов
- Fetch только новых: `UID FETCH lastUidNext:*`
- Быстрее и надёжнее

### 2. Message-ID для дедупликации
**Проблема:**
- Свои отправленные письма попадают в INBOX (через BCC)
- IDLE срабатывает на них
- Обрабатываются как входящие

**Решение (Delta Chat):**
- Каждое письмо получает уникальный Message-ID
- При отправке сохраняем Message-ID в БД
- При получении проверяем: если Message-ID уже обработан → пропускаем
- Дополнительная проверка `from == myEmail`

### 3. BCC to self для мульти-девайс
**Зачем:**
- Синхронизация отправленных сообщений между устройствами
- Отправил с телефона → письмо приходит на все устройства
- Все девайсы видят полную историю

**Как работает:**
- При отправке добавляем себя в BCC
- Письмо приходит обратно в INBOX
- Проверяем Message-ID → пропускаем дубликат
- Но сохраняем для других устройств

### 4. Батчинг для большого количества писем
**Зачем:**
- Если долго не заходил → накопилось много писем
- Fetch всех сразу может упасть (команда слишком длинная)
- Fetch батчами по 50 писем безопаснее

**Как работает:**
```dart
if (totalNew > 50) {
  // Fetch по 50 писем за раз
  for (batch in batches) {
    fetch(batch)
    pause 100ms
  }
}
```

### 5. UIDVALIDITY для проверки пересоздания ящика
- Если UIDVALIDITY изменился → ящик пересоздан на сервере
- Все UID сбрасываются
- Нужно начать синхронизацию заново

### 4. BODY.PEEK[] вместо RFC822
**Проблема RFC822:**
- Автоматически помечает письма как \Seen
- Теряем контроль над флагом
- Письма помечаются ДО обработки

**Решение BODY.PEEK[] (Delta Chat):**
- НЕ помечает как \Seen автоматически
- Мы сами контролируем когда пометить
- Помечаем только после успешной обработки

### 5. IDLE + NOOP
- IDLE работает 29 минут (RFC рекомендует до 30)
- После DONE делается NOOP для проверки соединения
- Если NOOP падает → переподключение

## Основной флоу

### Отправка
```
1. Создаём Message-ID: <timestamp.random@domain>
2. Шифруем текст
3. Добавляем BCC to self (для мульти-девайс)
4. Отправляем через SMTP
5. Сохраняем в БД с Message-ID
6. Добавляем Message-ID в processed_message_ids
7. Письмо придёт обратно через BCC → будет отфильтровано по Message-ID
```

### Получение
```
1. IDLE уведомляет о новом письме (EXISTS увеличился)
2. Fetch проверяет UIDNEXT
3. Если UIDNEXT увеличился:
   - Если новых > 50 → fetch батчами по 50
   - Иначе → fetch все сразу
4. Проверяем Message-ID: если уже обработан → пропускаем (свои письма)
5. Проверяем from: если свой → пропускаем (дополнительная защита)
6. Расшифровываем и сохраняем
7. Помечаем как SEEN вручную (контролируем флаг)
8. Добавляем в processed_uids и processed_message_ids
```

### IDLE цикл
```
while (true) {
  IDLE 29 минут или до события
  DONE
  NOOP (проверка соединения)
  Если событие → уведомить UI
  Пауза 100мс
  Повтор
}
```

## Защита от проблем

### Дубликаты
- Message-ID дедупликация (основная) - фильтрует свои письма через BCC
- UID дедупликация (дополнительная)
- Проверка `from == myEmail` (на всякий случай)

### Батчинг
- Если новых писем > 50 → fetch по 50 за раз
- Пауза 100мс между батчами
- Защита от перегрузки при большом количестве писем

### Мульти-девайс
- BCC to self включён по умолчанию
- Свои письма приходят обратно
- Фильтруются по Message-ID
- Все устройства видят полную историю

### IDLE стабильность
- 29 минут (не 28, не 30)
- NOOP после DONE
- Проверка EXISTS увеличился
- Переподключение при ошибках

### SMTP стабильность
- Новый клиент каждый раз
- Всегда закрывается

### Старые/битые письма
- UID = 0 пропускаются
- Не-JSON пропускаются
- Ошибки расшифровки не показываются

## Почему это работает

**Delta Chat использует этот подход годами:**
- UIDNEXT - стандарт RFC 3501
- Message-ID - стандарт RFC 5322
- IDLE 29 минут - best practice
- NOOP после DONE - проверка соединения

**Наш код теперь следует тем же принципам.**
