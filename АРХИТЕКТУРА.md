# Архитектура приложения (Production-Ready)

## Обзор

Приложение использует чистую архитектуру с разделением ответственности:
- **UI (Screens)** - только отображение и взаимодействие с пользователем
- **Services** - вся бизнес-логика и работа с данными
- **Storage** - работа с базой данных

## Структура сервисов

### 1. ChatService (главный координатор)
**Файл**: `lib/services/chat_service.dart`

**Ответственность**:
- Единая точка входа для UI
- Связывает EmailService и MessageService
- Управляет жизненным циклом подключения

**Методы**:
- `initialize()` - инициализация (загрузка ключей, подключение к IMAP)
- `addUICallback()` - регистрация callback для обновления UI
- `removeUICallback()` - удаление callback
- `sendMessage()` - отправка сообщения
- `disconnect()` - отключение

**Использование**:
```dart
final chatService = ChatService(email: email, password: password);
await chatService.initialize();

// Регистрируем callback для обновления UI
chatService.addUICallback(() {
  if (mounted) {
    _loadMessages();
  }
});
```

---

### 2. MessageService (обработка сообщений)
**Файл**: `lib/services/message_service.dart`

**Ответственность**:
- Обработка входящих писем
- Расшифровка сообщений
- Сохранение в БД
- Уведомление UI через callbacks

**Методы**:
- `processNewMessages()` - обработка списка новых писем
- `addUICallback()` - регистрация callback
- `removeUICallback()` - удаление callback

**Внутренняя логика**:
- `_processMessage()` - обработка одного письма
- `_handleInvite()` - обработка приглашения
- `_handleReadReceipt()` - обработка read receipt
- `_handleTextMessage()` - обработка текстового сообщения

---

### 3. AccountService (управление аккаунтами)
**Файл**: `lib/services/account_service.dart`

**Ответственность**:
- Генерация ключей
- Загрузка ключей из БД
- Сохранение аккаунтов

**Методы**:
- `loadOrGenerateAccount()` - загрузка или генерация ключей

**Возвращает**: `AccountData` с email, keyPair, publicKeyHex

---

### 4. EmailService (транспорт)
**Файл**: `lib/services/email_service.dart`

**Ответственность**:
- IMAP подключение и IDLE
- SMTP отправка
- Fetch новых писем
- Уведомление о новых письмах

**Методы**:
- `connectImap()` - подключение к IMAP
- `fetchNewMessages()` - получение новых писем
- `sendMessage()` - отправка письма
- `setMessageProcessor()` - установка обработчика сообщений
- `disconnect()` - отключение

---

### 5. CryptoService (шифрование)
**Файл**: `lib/services/crypto_service.dart`

**Ответственность**:
- Генерация ключей
- Шифрование/расшифровка
- Экспорт/импорт ключей

---

### 6. StorageService (база данных)
**Файл**: `lib/services/storage_service.dart`

**Ответственность**:
- Сохранение/загрузка сообщений
- Сохранение/загрузка контактов
- Сохранение/загрузка аккаунтов
- Отслеживание обработанных UID

## Поток данных

### Инициализация приложения

```
ChatListScreen
    ↓
ChatService.initialize()
    ↓
AccountService.loadOrGenerateAccount() → загрузка ключей
    ↓
MessageService (создание)
    ↓
EmailService (создание)
    ↓
EmailService.connectImap() → подключение к IMAP
    ↓
IDLE loop запущен
```

### Получение нового сообщения

```
IMAP IDLE событие
    ↓
EmailService._startIdleLoop()
    ↓
EmailService.fetchNewMessages() → fetch с сервера
    ↓
MessageService.processNewMessages() → обработка
    ↓
MessageService._processMessage() → расшифровка
    ↓
StorageService.saveMessage() → сохранение в БД
    ↓
MessageService._notifyUI() → уведомление UI
    ↓
ChatListScreen/ChatScreen callbacks → обновление UI
    ↓
StorageService.getMessages() → загрузка из БД
    ↓
setState() → отображение
```

### Отправка сообщения

```
ChatScreen._handleSendPressed()
    ↓
StorageService.saveMessage() → сохранение в БД (status: sending)
    ↓
CryptoService.encryptMessage() → шифрование
    ↓
ChatService.sendMessage() → отправка через EmailService
    ↓
EmailService.sendMessage() → SMTP отправка
    ↓
StorageService.updateMessageStatus() → обновление статуса (status: sent)
    ↓
setState() → обновление UI
```

## Правила архитектуры

### ✅ ПРАВИЛЬНО

1. **UI экраны**:
   - Только загружают данные из БД
   - Регистрируют callbacks для обновления
   - Вызывают методы ChatService для действий

2. **Сервисы**:
   - Вся бизнес-логика в сервисах
   - Обработка сообщений в MessageService
   - Транспорт в EmailService
   - UI только уведомляется через callbacks

3. **Единообразие**:
   - Все экраны работают одинаково
   - Все загружают из БД
   - Все обновляются через callbacks

### ❌ НЕПРАВИЛЬНО

1. **НЕ дублировать логику**:
   - ❌ Обработка сообщений в UI
   - ❌ Разная логика в ChatListScreen и ChatScreen
   - ❌ Прямые вызовы EmailService из UI

2. **НЕ смешивать ответственность**:
   - ❌ Бизнес-логика в UI
   - ❌ UI логика в сервисах
   - ❌ Транспорт в обработчиках сообщений

## Преимущества новой архитектуры

1. **Чистота кода**: Каждый компонент отвечает за свою задачу
2. **Тестируемость**: Сервисы можно тестировать независимо
3. **Масштабируемость**: Легко добавлять новые функции
4. **Надёжность**: Единая точка обработки сообщений
5. **Производительность**: Нет дублирования обработки
6. **Поддерживаемость**: Понятная структура кода

## Миграция со старой архитектуры

### Было (старая архитектура):
- ChatListScreen напрямую работал с EmailService
- ChatListScreen сам обрабатывал сообщения
- ChatScreen делал fetch и обработку
- Дублирование логики между экранами

### Стало (новая архитектура):
- ChatService - единая точка входа
- MessageService - вся обработка сообщений
- UI экраны - только отображение
- Нет дублирования логики
